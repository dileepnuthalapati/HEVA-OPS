# Heva One - Product Requirements Document

## Overview
Multi-tenant **Modular SaaS** business management system. Cloud backend (FastAPI + MongoDB), mobile-first frontend (React + Capacitor APK). Revenue model: **per-module monthly pricing + 0.3% commission on QR orders via Stripe Connect**.

## Universal App Architecture
ONE app (Capacitor), TWO device modes — "Split-Brain" routing:

| Mode | Device | Boot Screen | Experience |
|---|---|---|---|
| **Terminal (Kiosk)** | Store tablet | PIN Pad | Staff enter PIN -> POS or Clock-In toast |
| **Personal** | Staff phone | Email + Password | Heva Ops workspace (shifts, clock-in, swaps) |

## Email System (Apr 12, 2026)
**Provider:** Resend (API key in .env)
**Sender:** `noreply@hetupathways.com` (domain verified)

**Endpoints:**
- `POST /api/email/test` — Send test email to current admin (requires auth)
- `POST /api/email/daily-summary/send` — Send yesterday's summary to current business admin
- `POST /api/email/daily-summary/send-all` — Platform owner sends to ALL businesses
- `POST /api/email/trial-reminders/send` — Platform owner triggers trial expiry checks (7d, 3d, 1d)

**Daily Summary includes:** Total revenue, order count, cash/card breakdown, avg order value, top 5 sellers, staff performance
**Trial Reminders:** Sent at 7, 3, 1, 0 days before `trial_end_date` on restaurant document

**For production:** Set up Railway cron job to call `/api/email/daily-summary/send-all` daily at 8 AM

55. **Portal Race + Logo + Push Crash + Mobile Heading (Feb 11, 2026)** — Follow-up from production feedback: iteration 54's Sheet-close fix didn't solve the real "Workforce freeze" bug. True RCA (via troubleshoot agent) was a Radix DropdownMenu portal race in the Workspace Switcher:
   - **(a) Workspace switcher portal race** — the DropdownMenu was uncontrolled; clicking an item fired `navigate()` synchronously before the Radix dismissable layer could unmount, leaving an invisible `z-50` overlay on top of the new page that blocked every click (symptom: user had to fully close browser tab / APK to recover). Fix: made `DropdownMenu` controlled via `open`/`onOpenChange`, close it first, then navigate via `requestAnimationFrame(() => onSelect(key))`. Verified via Playwright: 0 lingering `[data-radix-popper-content-wrapper]` elements after switch; target page immediately interactive.
   - **(b) APK ↔ PWA logo sync** — regenerated `public/favicon.png`, `icon-192.png`, `icon-512.png`, `icon-maskable-512.png`, `apple-touch-icon.png` **from** the Android adaptive-icon foreground (`android/app/src/main/res/mipmap-xxxhdpi/ic_launcher_foreground.png`) so the PWA now shows the same "H1E" letterform the user likes. Non-maskable icons scale the foreground up 1.5x to compensate for Android launcher safe-zone padding; maskable keeps the 66% safe zone for launcher mask compatibility.
   - **(c) Push notifications crashing APK** — real root cause was the missing `POST_NOTIFICATIONS` permission on Android 13+ (required since API 33 — without it, `PushNotifications.register()` hard-crashes the activity). Added `<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />` to `AndroidManifest.xml`. Also restructured `push.js` to attach all listeners BEFORE calling `register()` (best practice — prevents the FCM token from firing before a listener is ready) and to handle `prompt-with-rationale` permission state.
   - **(d) Mobile page heading overlap** — the fixed mobile header now includes `env(safe-area-inset-top)` padding which made it taller than the `pt-16` (4rem) content offset used across every page. Added a targeted CSS rule in `App.css` that scopes to the `.pt-16.md\\:pt-6` pattern so all pages automatically pad by `calc(4rem + env(safe-area-inset-top))` on mobile — without touching 20 page files individually.
   - **(e) Attendance page week-day range** — the default filter on `/workforce/attendance` was "last 7 days from today" (`now − 7 → now`) which ignored `week_start_day`. Now defaults to the current work-week honouring `week_start_day` (Sun/Mon/Sat start) loaded from restaurant settings; if the user manually edits the date inputs that edit is preserved. Timesheets and Shift Scheduler were verified already correct.

57. **Live Customer Fixes — Session Restore, Feature Staleness, Rota Persona, APK Backup, Safari/APK Click Lock (Feb 11, 2026)** — Another urgent prod-feedback round, real root causes only:
   - **(a) APK auto-logs in as old user after uninstall/reinstall.** Actual root cause: `android:allowBackup="true"` in AndroidManifest → Google Auto-Backup silently saves the WebView's localStorage (including `auth_token`, `user`) to Google Drive and restores it on reinstall. Fix: set `android:allowBackup="false"` + added `@xml/backup_rules` (exclude all domains) and `@xml/data_extraction_rules` for Android 12+ (excludes both cloud-backup AND device-transfer). Belt-and-braces on the JS side: `AuthContext` now verifies the session via `/auth/me` on mount — a stale restored token 401s, we wipe localStorage and redirect to login.

   **UPDATE (same day): Safari/APK click-lock on switcher — REAL root cause found.** User kept reporting "on Safari + APK, the new page loads but nothing is clickable, I have to click the module a second time." Previous iter-55/56 fixes (rAF defer, pending-navigation useEffect) hid the symptom on Chromium but Safari/Android WebView still failed. Actual cause: Radix `<DropdownMenu>` renders in **modal mode by default**, which sets `pointer-events: none` on `<body>` during the open/close animation (~150-300 ms on Safari). On desktop browsers, the rapid transition is imperceptible; on Safari + Android WebView the lock outlives the navigation and the destination page stays frozen until the user's second tap unfreezes it. Fix: **`modal={false}` on the Workspace Switcher DropdownMenu** — removes the body pointer-events lock + focus trap entirely. Verified via Playwright: 5 consecutive POS↔Workforce switches all show `body.pointer-events=auto`, 0 portals, H1 element-at-point check passes on every destination immediately.
   - **(b) "Hard refresh needed after enabling modules."** Real root cause: features are loaded at login, cached in localStorage, and never refreshed until next login. When a platform admin enables a new module server-side, the logged-in user's cached `features` stays stale → Sidebar/workspace-switcher still shows the old set. Fix: added a polling `useEffect` in `AuthContext` that calls `/auth/features` on `window.focus`, `document.visibilitychange → visible`, and every 60 s as a safety net. **Upgrade (same day):** also fires the refresh on every React Router route change via `refreshFeatures` exposed through context and called from `Sidebar`'s `useEffect([location.pathname])`. Module-level throttle caps the API to at most 1 call per 3 seconds so rapid navigation doesn't spam the server. Effective result: stale-feature window is "until the user next clicks anything" — effectively instant. The 60 s interval is the safety net for idle tabs, NOT the app load time.
   - **(c) "Workforce-only businesses shouldn't see POS/KDS in the switcher."** Removed the `locked` workspace array entirely — for a healthcare/retail customer who bought Workforce-only, the switcher is now a single static header (no dropdown, no upsell). Enabled workspaces still render normally for multi-module customers.
   - **(d) "manage_rota persona does nothing."** Real root cause: routes `/workforce/shifts|attendance|timesheets` were `restaurantAdminOnly` — ignored the capability. Fixes: (i) `ProtectedRoute` now accepts a `capability` prop and admits staff who have it; (ii) applied `capability="workforce.manage_rota"` to the three workforce admin routes; (iii) backend: new `require_rota_manager` dependency applied to `shifts.*`, `/attendance/live`, `/restaurant/staff` (list only), `/timesheets/summary`. Admin-only ops (approve, reject, unlock, force-close, staff CRUD) still require_admin; (iv) `User` model now exposes `capabilities` via `/auth/me` so the frontend can refresh them without a relogin; (v) Sidebar swaps in the admin Workforce workspace for a manage_rota staff so they see "Shift Scheduler / Attendance / Timesheets" (not staff "My Shifts / Clock / My Pay"); (vi) Dashboard link hidden for non-admin — rota-manager is a team lead, not a business admin; (vii) Timesheets page shows "View only" label in the Action column instead of Approve/Reject buttons; (viii) AttendancePage "Force out" button remains admin-only.
   - **(e) "Powered by Heva ONE" → "Powered by Hetu Pathways"** in Login footer, Receipt template preview (RestaurantSettings), and actual printed-receipt footer (`receiptGenerator.js`).
   - **(f) Push APK still crashing when tapping Enable.** Additional hardening on top of the iter-55 `isPluginAvailable` gate: wrapped every individual `PushNotifications.*` native call in its own try/catch (so one crashing step doesn't kill the whole flow), added a 10 s race-timeout around `register()` so the UI can recover even if native Firebase init hangs, and returns clearer messages ("Run `npx cap sync android` locally, then rebuild the APK" when plugin is missing; "Notification service did not respond" on timeout).
